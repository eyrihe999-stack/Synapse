# 极端场景与过度防御清单

> 本文件定义"几乎不可能触发的极端场景"。代码中对这些场景的防御视为冗余，测试中对这些场景的覆盖视为低价值。

## 代码侧：禁止做过度防御的场景

以下代码模式视为冗余，应标记为问题并删除：

1. **整数溢出检查**（如 `if balance > math.MaxInt64-delta`）——业务数值范围远小于 int64 上限
2. **nil 依赖防御**（如 `if s.logger != nil { s.logger.Error(...) }`）——构造函数已保证依赖非 nil
3. **内部数据一致性断言**（如检查 map 中是否存在某 key、slice 索引是否越界）——上游逻辑已保证一致性
4. **DB 连接级错误的独立处理**（如 `if err == sql.ErrConnDone`）——全局故障由基础设施监控兜底
5. **不可能的枚举 default 分支做复杂错误处理**——switch 已穷举所有 case 时，default 留空并记一条 warn 日志即可
6. **上游 binding 已校验的参数在 service 层再次校验**——binding tag 已保证
7. **不可能的状态组合**——数据模型和业务逻辑已保证不存在

## 测试侧：禁止新增的测试类型

以下类型的测试不得新增，已有的标记为冗余：

1. **纯 DB 连接/磁盘错误测试**——全局故障由基础设施兜底
2. **整数溢出/极值测试**——业务数值范围远小于 int64 上限
3. **nil 安全测试**——构造函数保证非 nil
4. **内部数据一致性断言测试**——代码 bug 而非用户输入导致的场景
5. **上游已保证的参数校验测试**——binding 层已校验
6. **返回值字段完整性测试**——DTO 映射正确性不需要逐字段测试
7. **并发极端压力测试**（100+ goroutine）——5-10 个足以验证并发安全
8. **不可能的状态组合测试**——数据模型已保证不存在

## 不算过度防御的场景（保留）

以下场景是合理的防御，不应删除：

- 用户可通过 API 直接触发的输入校验
- 并发场景下的竞态防护（FOR UPDATE、条件 WHERE、乐观锁）
- 外部 API 返回异常值的处理
- 跨模块调用的参数校验（调用方不在本模块控制范围内）
