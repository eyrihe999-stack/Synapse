# 代码规模约束

> 本文件定义文件行数、函数行数、测试文件规模的统一阈值。
>
> **这些阈值不是硬性限制。** 复杂事务函数、switch/case 密集的错误映射、表驱动初始化等场景，强行拆分反而降低可读性和逻辑连贯性，可适当超出。超出时应在审计报告中标注原因，不自动标记为问题。

## 源文件

- 单个 `.go` 文件不超过 **400 行**
- 超过时按职责拆分（如 `org_service.go`、`member_service.go`、`role_service.go`）
- 文件名必须有语义，禁止 `service2.go`、`utils.go`、`helpers.go` 等无语义命名
- **不算超标的例外**：model 定义文件（多 struct 集中）、路由注册文件、索引定义文件

## 函数

- 单个函数不超过 **200 行**（不含空行和纯注释行）
- 超过时应评估是否能提取子步骤为独立函数
- **不算超标的例外**：
  - 函数体主要由 switch/case 或 if/else 分支组成，且每个分支逻辑简单独立（如错误映射、状态机转换）
  - 表驱动的初始化代码（如索引定义、配置声明）
  - 复杂事务函数因逻辑连贯性需要保持在一个函数内

## 测试文件

- 单个 `_test.go` 文件不超过 **400 行**
- 单个 `_test.go` 文件不超过 **20 个顶层 Test 函数**
- 超过时按被测领域拆分，每组生成 `{source_name}_test.go`
- 共享的 mock setup / helper 保留在 `helpers_test.go`
- **不算超标的例外**：表驱动测试（单个 Test 函数内有大量子测试但顶层函数数未超标）

## 未导出函数注释

- 超过 **10 行**或包含业务规则的未导出函数应有注释
