# 冗余代码与过度防御规范

> 统一的冗余代码检查标准。

---

## 一、未使用代码

### 1.1 未使用的导出函数/方法

对模块下每个导出函数和方法：
1. 用 Grep 在**整个项目**（`internal/` + `cmd/` + `pkg/`）中搜索其名称
2. 只有定义处有匹配、无调用者 → 标记为「未使用」

### 1.2 未使用的未导出函数/方法

对模块下每个未导出函数和方法：
1. 用 Grep 在**同 package** 内搜索其名称
2. 只有定义处有匹配、无调用者 → 标记为「未使用」

### 1.3 未使用的类型/常量/变量

对 `const.go`、`errors.go`、`model/` 中定义的导出类型、常量、变量：
1. 用 Grep 在整个项目中搜索
2. 只有定义处有匹配 → 标记为「未使用」

### 排除规则

- 接口实现方法不算未使用
- `init()` 函数不算未使用
- `_test.go` 中通过 `export_test.go` 导出的内部函数不算未使用
- 宁可漏报不可误删：不确定是否有间接引用（反射、interface、`go:generate`）的，放入「不确定项」

## 二、重复逻辑

- 找出函数体高度相似（>80% 相同逻辑）的函数对
- 找出被复制粘贴的代码块
- 建议合并或提取公共函数

**核心原则：不要牺牲可读性强行报抽象公共逻辑。** a little copying is better than a little dependency。

**常见的排除场景**：
- CRUD 模板代码
- 错误处理模式（`if err != nil` + 日志 + wrap error）
- 数据转换/映射代码（model → DTO 的字段赋值）
- 任何「模式相同但参数/类型/上下文不同」的代码

---

## 三、过度防御

对不可能发生的场景做了防护代码。判断必须结合**真实业务场景**。

**判断范围**：不能只看当前函数内部，必须沿着完整调用链来判断。

### 检查项

1. **上游已保证的校验**：追踪参数来源，上游已保证条件不成立 → 冗余
2. **不可能的业务场景**：结合真实业务流程判断该分支是否可能被触发
3. **重复的错误判断**：上游已区分过错误类型，下游又重复判断

详细清单见 [extreme-scenarios.md](extreme-scenarios.md)。

---

## 四、注意事项

- 问题等级为 P2，删除时必须确认没有外部调用者，不确定时放入「不确定项」
- 测试侧删除后必须确保 `go vet` + `go test` 通过
